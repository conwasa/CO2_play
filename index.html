About

Readings are taken every 10 minutes.

Some of the data has structure. Why? The sensor is in a street of 1920s houses with one or two cars or vans per house. At the time of writing there are daily peaks in the morning and early evening. Its not the cars. Its winter and plant growth is stunted. Its cold outside.

Why are there spikes in the data?

The houses were fitted with central heating boilers in the 1970s. The boilers vent through house walls at ground level and mostly run on timers to burn twice a day. That causes the spikes. The CO2 blows away when its windy, resulting in periods without spikes.

Development Notes

The hardware is a consumer level indoor air quality meter (precision: ±75ppm + 5% of measured value) in a weatherproof enclosure connected to a laptop via a 30ft (10m) USB extension cable. The sensor is at ground level.

Back-end: Every 10 minutes, a batch file copies daily CSV files, produced by proprietary logging software into one file, which is processed to produce one CSV file for each graph. The CSV files are copied to a publicly visible Cloud Storage bucket. The backend was developed on a Windows 7 laptop and ran there from November 2018 to February 2019, when it was moved to a stable "production" laptop running Linux Debian 9.

Front-end: The graphs and the outages table (below) are rendered with Google's Visualisation API, Statistics and the sensor's location () are rendered using Handlebars. A CORS problem accessing storage.googleapis.com/uk_bn3_co2/ from Github was bypassed using bypasscors.herokuapp.com until it broken in November 2019, when it was replaced with cors-anywhere.herokuapp.com.

Source code is here: github.com/conwasa/CO2_play. The scripts are:

  • process_CO2_data.bat   Windows/DOS script to copy files and run scripts every 10 minutes
  • process_csv.py         Python script to create CSV files used by this HTML

  • monitor_sensor_connection.py sounds an alarm if data older than 15 minutes
